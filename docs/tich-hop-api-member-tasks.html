API member-tasks — tra cứu task của một nhân sự

Cập nhật: 20/07/2026 · Người viết: Giang (Easygoing) · Phiên bản: 1.0

Giới thiệu

API này trả về danh sách task mà một nhân sự cụ thể đang được giao trong hệ thống Bot Work Tracker của Easygoing: tiêu đề, trạng thái, độ ưu tiên, dự án / tính năng / sprint mà task thuộc về, hạn chót và tiến độ checklist. Tài liệu dành cho đội kỹ thuật bên thứ ba cần đọc dữ liệu này từ hệ thống khác.

  • API chỉ đọc (read-only) — không có thao tác ghi.
  • Tra cứu một người mỗi lượt gọi, định danh bằng email, id nội bộ hoặc Discord id.
  • Có hai đường gọi tương đương nhau về dữ liệu — chọn một trong hai (xem bảng dưới).
Đường gọiGiao thứcĐộ trễ đo thực tếNên dùng khi
KHUYÊN DÙNG RPC PostgREST POST + JSON body ~0,3 giây, ổn định (không cold start) Mọi tích hợp server-to-server, cần phản hồi nhanh

Cách 1 — RPC PostgREST KHUYÊN DÙNG

POST https://vlsskdwcfcmyubyrtwhn.supabase.co/rest/v1/rpc/member_tasks

Header bắt buộc:

apikey: sb_publishable_Bf6XLnvqxVvPP4lVao-Q5A_PDJTJRdG
Content-Type: application/json

Body là JSON — tối thiểu gồm p_key và đúng một định danh nhân sự:

{
  "p_key": "<API_KEY>",
  "p_email": "nhansu@easygoing.vn"
}
Khác REST thường Đường RPC luôn trả HTTP 200, kể cả khi sai key hay không tìm thấy người. Ứng dụng phải kiểm tra trường error trong body trước khi đọc dữ liệu (xem mục Mã lỗi).

Ví dụ curl

curl -X POST "https://vlsskdwcfcmyubyrtwhn.supabase.co/rest/v1/rpc/member_tasks" \
  -H "apikey: sb_publishable_Bf6XLnvqxVvPP4lVao-Q5A_PDJTJRdG" \
  -H "Content-Type: application/json" \
  -d '{"p_key": "<API_KEY>", "p_email": "nhansu@easygoing.vn", "p_status": "all"}'

Ví dụ JavaScript (fetch)

const res = await fetch(
  'https://vlsskdwcfcmyubyrtwhn.supabase.co/rest/v1/rpc/member_tasks',
  {
    method: 'POST',
    headers: {
      apikey: 'sb_publishable_Bf6XLnvqxVvPP4lVao-Q5A_PDJTJRdG',
      'Content-Type': 'application/json',
    },
    body: JSON.stringify({ p_key: API_KEY, p_email: 'nhansu@easygoing.vn' }),
  },
);
const data = await res.json();
if (data.error) throw new Error(data.error);  // unauthorized / member_not_found / …
console.log(data.member.displayName, data.count, data.tasks);

Cấu hình Postman

  1. Method POST, dán URL ở trên.
  2. Tab Headers: thêm apikey (publishable key) và Content-Type: application/json.
  3. Tab Body → chọn raw / JSON → dán body ví dụ, thay <API_KEY> bằng key được cấp.
  4. Bấm Send — phản hồi ~0,3 giây.

Cách 2 — Edge Function THAY THẾ

GET https://vlsskdwcfcmyubyrtwhn.supabase.co/functions/v1/member-tasks

Header bắt buộc (không cần apikey ở đường này):

x-api-key: <API_KEY>

Tham số truyền qua query string, ví dụ:

curl "https://vlsskdwcfcmyubyrtwhn.supabase.co/functions/v1/member-tasks?email=nhansu@easygoing.vn&status=all" \
  -H "x-api-key: <API_KEY>"

Đường này trả mã lỗi HTTP thật (401 sai key, 404 không tìm thấy người, 400 tham số sai, 405 sai method) và bật CORS (Access-Control-Allow-Origin: *) nên gọi được từ trình duyệt. Đổi lại, độ trễ cao hơn đường RPC và phát gọi đầu tiên sau một lúc không ai gọi sẽ mất ~2 giây (cold start).

Mẹo Postman Dán URL cẩn thận — thừa một dấu cách giữa đường dẫn và ? là path đổi và server báo NOT_FOUND. Nếu gặp lỗi này, xoá ô URL và dán lại thành một dòng liền.

Thứ tự sắp xếp tasks được sắp sẵn: việc đang chạy trước (in_progress → review → todo → done), trong cùng trạng thái thì hạn chót gần nhất lên đầu (task không có hạn xếp cuối). Mọi mốc thời gian là ISO 8601 múi giờ UTC (+00:00) — bên tích hợp tự quy đổi sang giờ Việt Nam (UTC+7) khi hiển thị.

Với đường RPC, quy tắc an toàn là: nếu body có trường error thì coi là thất bại, ngược lại đọc member / tasks.

Lưu ý tích hợp

  • Hiệu năng: đường RPC phản hồi ~0,3 giây từ Việt Nam. Nếu gọi nhiều lần liên tiếp, giữ kết nối HTTP keep-alive (mặc định của đa số HTTP client) để tiết kiệm thêm ~0,15 giây bắt tay TLS mỗi request.
  • Tần suất: API phục vụ tra cứu, không có rate limit riêng — nhưng vui lòng không polling dày hơn 1 lần/phút cho mỗi nhân sự; dữ liệu task không đổi nhanh hơn thế.
  • Cache: nếu hiển thị dashboard, cache kết quả 30–60 giây là hợp lý.
  • Danh sách nhân sự: API không có endpoint liệt kê toàn bộ nhân sự — danh sách email cần theo dõi do hai bên thống nhất trước.
  • Thay đổi format: các trường hiện có sẽ không bị đổi tên/xoá mà không báo trước; trường mới có thể được thêm — parser nên bỏ qua trường lạ thay vì báo lỗi.
  • Đầu mối hỗ trợ: mọi vấn đề về key (cấp lại, thu hồi), thêm nhân sự, hoặc lỗi API — liên hệ Easygoing (Giang).
Checklist tích hợp nhanh ① Nhận API key qua kênh riêng → ② Gọi thử RPC với email được cấp → ③ Kiểm tra trường error trong body → ④ Map tasks[] vào hệ thống của bạn → ⑤ Đặt cache/polling hợp lý. Xong.
