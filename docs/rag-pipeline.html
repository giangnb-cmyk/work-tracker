Quy trình RAG của bot

Cập nhật: 21/07/2026 · Người viết: giangnb · Dự án: Bot Work Tracker (bot/)

Tổng quan

RAG (Retrieval-Augmented Generation) là cách bot Discord trả lời câu hỏi về nội dung tài liệu (spec, hướng dẫn, quy trình, meeting notes…). Thay vì để mô hình "bịa", bot đi tìm các đoạn văn liên quan trong kho rồi mới tổng hợp câu trả lời dựa trên đúng các đoạn đó, kèm trích nguồn.

Hệ thống gồm hai luồng tách rời:

Luồng 1 · offline 📥 Nạp tài liệu Chạy nền, tự động 9h/ngày. Đọc tài liệu → cắt đoạn → tạo embedding → lưu vào kho vector. Người dùng không thấy.
Luồng 2 · realtime 💬 Hỏi tài liệu Khi bị @tag trên Discord. Biến câu hỏi thành vector → tìm đoạn gần nhất trong kho → Claude tổng hợp trả lời.

Cả hai luồng dùng chung một mô hình embedding (bge-m3) và chung một kho (bảng documents trên Supabase + pgvector) — đó là điểm mấu chốt để "câu hỏi" và "tài liệu" so khớp được với nhau.

Luồng 1 — Nạp tài liệu (offline)

Điều phối bởi bot.py → daily_rag_sync(), chạy lúc 9h sáng (Asia/Ho_Chi_Minh) mỗi ngày nếu bật rag_sync_enabled. Gồm 3 phần tách lỗi (một phần hỏng không làm gãy phần còn lại):

Điểm kích hoạt ⏰ daily_rag_sync — 9h mỗi ngày bot.py, chỉ chạy khi rag_sync_enabled = true. Mỗi phần có giới hạn thời gian riêng (drive_ingest tới 3 giờ).
▼ chạy 3 phần
Phần (a) 📂 sync_docs.py Tài liệu trong thư mục docs/ + các link ngoài đã khai báo ("tài liệu chung").
Phần (b) 🗂️ drive_catalog.py Danh mục Google Drive: chỉ ghi file NẰM Ở ĐÂU (trả lời "file ở đâu"), không đọc ruột.
Phần (c) 📄 drive_ingest.py Đọc RUỘT (nội dung) file trên Drive → đây là phần chính của RAG (xem chi tiết bên dưới).

Chi tiết phần (c) — drive_ingest.py

Đây là nơi tài liệu Drive được biến thành vector và lưu vào kho:

Nguồn ☁️ Drive API — liệt kê file Mọi file được share cho service account (drive_gateway.list_documents).
▼
Lọc 🧹 Bỏ file không cần rag_drive_skip: bỏ folder csv_config, tên chứa localization (toàn chuỗi/số, nạp vào chỉ nhiễu kho).
▼
Tăng dần ⏭️ Bỏ qua file không đổi So modifiedTime với source_version đã lưu — chỉ nạp lại file vừa sửa. Embedding local rất chậm nên không nạp lại cả kho mỗi ngày.
▼ với mỗi file cần nạp
Đọc 📖 Tải/export → doc_reader.read_sections Tải file về (Sheet lớn thì đọc qua Sheets API), tách thành các "mục" (section / tab).
▼
Cắt đoạn ✂️ chunk_text → build_pairs Cắt mỗi mục thành đoạn ~800 ký tự, gối đầu 120 ký tự giữa 2 đoạn để giữ ngữ cảnh.
▼
Embedding 🧠 embed_batch — bge-m3 qua Ollama Mỗi đoạn → 1 vector 1024 chiều. Gọi tuần tự (Ollama xử lý 1 prompt/lần) — đây là bước tốn thời gian nhất (~3s/đoạn).
▼
Ghi kho 💾 doc_repo.insert_chunks → bảng documents Lưu mỗi đoạn kèm: source, section, chunk_index, content, embedding(1024), source_url (deep link — Google Sheets trỏ đúng tab #gid), source_version, drive_folder_ids (id folder tổ tiên).
▼ sau khi duyệt hết file
Dọn 🗑️ Prune — gỡ nguồn đã mất Xoá khỏi kho những nguồn "Drive: …" không còn trên Drive (hoặc vừa bị lọc bỏ).
Vì sao nạp "tăng dần"? Kho hiện có khoảng 2.4 nghìn đoạn. Embedding chạy cục bộ trên máy bot nên rất chậm; nạp lại toàn bộ mỗi ngày là bất khả thi (nhiều giờ). Vì vậy chỉ file có modifiedTime thay đổi mới được đọc & embedding lại.

Kho lưu trữ — bảng documents

Nằm giữa hai luồng: luồng nạp ghi vào, luồng hỏi đọc ra. Là một bảng Postgres trên Supabase có cột embedding kiểu vector(1024) nhờ extension pgvector — cho phép tìm "đoạn gần nghĩa nhất" bằng khoảng cách cosine.

CộtÝ nghĩa
sourceTên nguồn (vd Drive: <tên file>).
section · chunk_indexMục/tab và số thứ tự đoạn trong nguồn.
contentNội dung đoạn văn (text gốc để hiển thị/trích).
embeddingvector(1024) — "dấu vân tay" ngữ nghĩa của đoạn, dùng để so khớp.
source_urlLink mở đúng chỗ (Google Sheets trỏ đúng tab qua #gid).
source_versionmodifiedTime của file — để nạp tăng dần.
drive_folder_idsTập id folder tổ tiên của file — để lọc tài liệu theo folder cho member.
project_idGắn theo dự án; NULL = tài liệu chung (dùng cho mọi dự án).

Luồng 2 — Hỏi tài liệu (realtime)

Xảy ra ngay khi có người @tag bot trên Discord và hỏi về nội dung tài liệu:

Đầu vào 💬 User @tag bot trên Discord bot.py nhận tin, truyền câu hỏi + BOT_SENDER_ID (discord id người hỏi) sang Claude CLI kèm bộ "hints".
▼
Điều phối 🤖 Claude CLI nhận diện "câu hỏi tài liệu" Theo hint, Claude quyết định chạy skill tìm tài liệu thay vì tra task/sprint.
▼ chạy skill
Skill 🔎 doc_search.py "<câu hỏi>" embed_text(câu hỏi) qua bge-m3 → 1 vector 1024 chiều (cùng "không gian" với tài liệu đã nạp).
▼
Tìm trong kho 📐 RPC match_documents (pgvector) So cosine giữa vector câu hỏi và mọi embedding, lấy top-12 đoạn gần nhất. Tham số: filter_project, filter_folder.
▼
Phân quyền 🔐 Lọc theo vai trò người hỏi Admin/owner: thấy tất cả. Member: chỉ folder cho phép (filter_folder) + ẩn nguồn nhạy cảm. Không nhận diện được → coi là member (khoá chặt). Xem mục dưới.
▼
Kết quả thô 📑 Các đoạn + 🔗 nguồn + 📁 link folder In ra cho Claude: từng đoạn kèm source_url và link thư mục tài liệu.
▼
Trả lời ✍️ Claude tổng hợp & gửi Discord Trả lời chỉ dựa trên các đoạn đó, trích nguồn [1][2]…, đính kèm link — rồi gửi lại kênh Discord.
Điểm mấu chốt Câu hỏi và tài liệu được embedding bằng cùng mô hình bge-m3, nên hai vector nằm chung "không gian nghĩa" và đo được độ gần. Đổi mô hình embedding thì phải nạp lại toàn bộ kho.

Phân quyền tài liệu (gate)

Danh tính người hỏi lấy từ BOT_SENDER_ID (id Discord thật, do bot.py đặt), khớp chính xác với profiles.discord_id. Từ đó quyết định phạm vi tài liệu:

Vai tròĐược thấy gìCơ chế
Admin / Owner Toàn bộ kho. filter_folder = NULL, không ẩn nguồn.
Member Chỉ tài liệu trong folder cấu hình (tính cả folder con) & không thấy nguồn nhạy cảm. Lọc ngay trong RPC qua filter_folder = rag_member_folder_id (so với drive_folder_ids) + chặn rag_member_hidden_sources.
Chưa link Discord Bị coi như member (khoá chặt). Không nhận diện được → mặc định phạm vi hẹp (fail-closed).
Cần backfill sau khi bật lọc folder Cột drive_folder_ids mới thêm, tài liệu nạp trước đó đang rỗng. Phải chạy một lần python skills/drive_ingest.py --backfill-folders trên máy bot; nếu chưa, member sẽ không thấy tài liệu nào.

Ghi chú vận hành

  • Bật/tắt sync tự động: rag_sync_enabled trong bot/settings.json.
  • Chạy tay: python skills/drive_ingest.py (nạp ruột Drive), --dry-run để chỉ đếm, --force để nạp lại cả file không đổi.
  • Backfill folder: --backfill-folders (gắn drive_folder_ids, không embedding lại).
  • Điều kiện chạy: Ollama phải bật trên máy bot (ollama serve + ollama pull bge-m3), và folder Drive đã share cho service account.
  • Thử tìm: python skills/doc_search.py "<câu hỏi>".
Tách vai trò rõ ràng drive_catalog.py trả lời "file nằm ở đâu" (danh mục), còn drive_ingest.py trả lời "trong file viết gì" (ruột). Hai skill khác nhau, chạy cùng lịch.
